# Phase 1: Diagnostics Model

## Goal

Split hidden-drive diagnostics enough to support actionable UI while preserving current filtering behavior and existing family/chart diagnostics.

## Scope

- Update `tools/drive_comparison_client/calc/filtering.js`.
- Update hidden reason priority in `tools/drive_comparison_client/state/core.js` if new reason keys need ordering.
- Update existing reason labels in `tools/drive_comparison_client/chart/interaction.js` only as needed to keep current warnings coherent.
- Regenerate published output through `npm run build`.

## Non-Goals

- Do not add the new actionable banner in this phase.
- Do not add action buttons.
- Do not make hidden search matches visible on the chart.
- Do not redesign filter controls.

## Affected Files

- `tools/drive_comparison_client/calc/filtering.js`
- `tools/drive_comparison_client/state/core.js`
- `tools/drive_comparison_client/chart/interaction.js`
- Generated by build: `docs/index.html`, `docs/assets/js/**`
- This phase document

## Implementation Steps

1. Split `evaluateDriveVisibility(row)` reason generation so engine count, search, and family/category filters are distinguishable:
   - `thrusterCountFilter`
   - `searchFilter`
   - `familyFilter`
2. Keep visibility semantics unchanged: a row is visible only when no reasons are present.
3. Keep family diagnostics scoped to rows matching selected engine count and search term, so existing family warnings do not treat search as a warning source.
4. Add a hidden summary to `computeDriveDiagnostics()`:
   - `hiddenSummary.totalHidden`
   - `hiddenSummary.byReason`
   - `hiddenSummary.dominantReason`
5. Add a search summary to `computeDriveDiagnostics()`:
   - `active`
   - `term`
   - `totalMatches`
   - `visibleMatches`
   - `hiddenMatches`
   - `hiddenByReason`
   - `dominantReason`
   - `sampleNames`
6. For search summary, evaluate only rows matching the search term and exclude `searchFilter` from hidden reasons.
7. Add or update a single user-facing reason-label function used by existing warnings.
8. Preserve existing diagnostic return fields: `visibleRows`, `families`, and `zeroFamilies`.

## Acceptance Criteria

- Existing chart and table still render with the same visible-row behavior.
- `computeDriveDiagnostics()` returns both `hiddenSummary` and `searchSummary`.
- Search diagnostics can report matching hidden drives without treating the search itself as the hiding reason.
- Existing family warning text still renders and uses Acceleration terminology for `minTwr`.
- No generated files are edited directly.

## Manual Smoke Tests

1. Open the generated chart through the browser verifier target.
2. Set metric to total mass.
3. Raise minimum acceleration enough to hide drives.
4. Confirm the visible count changes and existing family/chart diagnostics still render.
5. Search for a known drive name and confirm the chart remains usable.

## Rollback Risks

- If `searchFilter` leaks into family warnings, existing warnings may become noisy during search.
- If hidden summaries count unselected families too aggressively, phase 2 could over-warn users.
- If reason priority is wrong, the future banner may recommend a less useful action.
