# Phase 3: Browser Verifier Coverage

## Goal

Add automated browser coverage for the actionable hidden-state banner, action buttons, and language behavior.

## Scope

- Update `tools/verify_drive_comparison_browser.mjs`.
- Add Playwright checks that manipulate state through existing debug helpers or DOM controls.
- Keep assertions focused on durable behavior and key phrases.
- Regenerate published output through `npm run build`.

## Non-Goals

- Do not add visual screenshot baselines.
- Do not assert exact full localized sentences.
- Do not broaden the verifier to unrelated workflows.
- Do not change app behavior except for tiny testability fixes if required.

## Affected Files

- `tools/verify_drive_comparison_browser.mjs`
- Potentially source files if a small testability defect is found
- Generated by build: `docs/index.html`, `docs/assets/js/**`
- This phase document

## Implementation Steps

1. Add a general hidden-warning test:
   - set English mode
   - metric `totalMassTons`
   - high `minTwr`
   - `showImpracticalCandidates = false`
   - assert banner visible, mentions acceleration, includes reset/show actions
2. Add a search-hidden test:
   - choose a drive name from data
   - set `searchTerm`
   - set filters/threshold so matches are hidden
   - assert banner says matches exist but are hidden and includes the term
3. Add action behavior checks:
   - reset acceleration threshold
   - show impractical candidates
   - clear search
4. Add localization checks:
   - English banner does not show Korean source text
   - Korean banner does not show English action labels for the tested actions
5. Keep tests self-resetting so later verifier checks start from defaults.

## Acceptance Criteria

- `npm run verify:browser` fails if the banner is missing for high-acceleration hidden states.
- `npm run verify:browser` fails if search-hidden matches are not explained.
- `npm run verify:browser` fails if action buttons mutate state without syncing controls.
- English and Korean banner/action labels are covered by key-phrase assertions.
- Existing verifier coverage remains green.

## Validation Commands

```powershell
npm run build
npm run verify
```

Optional focused rerun during development:

```powershell
npm run verify:browser
```

## Manual Smoke Tests

1. Run the full browser verifier and review failure output if any assertion fails.
2. Open the generated chart.
3. Reproduce the high-minimum-acceleration banner manually.
4. Reproduce a hidden search result manually.
5. Exercise reset acceleration, show impractical candidates, and clear search.
6. Switch English and Korean and confirm no obvious untranslated banner/action text remains.

## Rollback Risks

- Over-specific text assertions could create noisy future failures. Prefer key phrases and state checks.
- Browser tests can be slow; keep new checks near existing setup/reset blocks and avoid extra reloads.
- Choosing a brittle fixture drive could fail when catalog data changes. Select fixture rows dynamically from `DATA.drives`.
